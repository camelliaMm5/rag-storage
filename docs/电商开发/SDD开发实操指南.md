# SDD 开发实操指南

> 以电商平台项目为案例，记录从需求文档到代码上线的完整 SDD（Specification-Driven Development，规范驱动开发）实践过程与方法论提炼。

---

## 一、SDD 概念与现状

### 1.1 什么是 SDD

**Specification-Driven Development（SDD，规范驱动开发）** 是一种以"规范文档为主产物、代码为派生品"的软件工程方法论。核心理念是：**规范声明意图，代码兑现意图。** 规范不是写完就扔的"脚手架"，而是伴随系统全生命周期的"活文档"。

SDD 在学术界和工业界都有明确定义。根据 Microsoft Learn 的 SDD 培训体系，SDD 包含四个标准阶段：

```
Specify（规范）→ Plan（计划）→ Tasks（任务）→ Implement（实现）
```

- **Specify**：定义软件要做什么、为什么做（用户故事、验收标准、边界条件）
- **Plan**：决定怎么做（架构、技术栈、设计方法）
- **Tasks**：将计划拆解为可独立执行和验证的开发任务
- **Implement**：在规范和任务清单的指导下编写代码，持续验证

### 1.2 SDD 的三个层次

学术界（Piskala, 2025）将 SDD 分为三个层次，这个分级对理解"我们做到什么程度"非常有帮助：

| 层次 | 定义 | 规范的命运 | 典型场景 | 本项目 |
|------|------|-----------|---------|--------|
| **Spec-First** | 写代码前先写规范 | 代码写完后规范可能被丢弃 | 原型、一次性功能 | — |
| **Spec-Anchored** | 规范作为"活文档"伴随代码全生命周期 | 测试保持规范与代码同步 | 大多数生产系统 | ✅ **本项目定位** |
| **Spec-as-Source** | 规范是唯一人工编辑的产物，代码完全自动生成 | 规范即源码 | API 生成（OpenAPI）、嵌入式系统（Simulink） | 未来方向 |

本项目处于 **Spec-Anchored** 层：需求文档和技术方案在代码编写前已完成，开发过程中持续对照文档验证行为，Bug 修复时回溯文档确认正确行为，测试脚本直接翻译自文档中的验收规则。

### 1.3 SDD 的行业现状（2024-2026）

SDD 在过去两年迎来了爆发式关注，直接原因是 **LLM（大语言模型）能力的跃升使得"规范到代码的翻译"变得可靠**：

| 时间 | 事件 | 意义 |
|------|------|------|
| 2024 年 10 月 | Claude 3.5 Sonnet 发布 | 长周期 Agent 编码工作流首次"积累正确而非积累错误" |
| 2025 年 7 月 | **AWS Kiro** 公测 | 第一个"规范优先"的 AI 编程 IDE，内置 Specify → Plan → Design → Implement 结构化阶段 |
| 2025 年 9 月 | **GitHub Spec Kit** 开源 | MIT 许可的 CLI 工具，定义了 `/specify`、`/plan`、`/tasks`、`/implement` 标准 SDD 命令 |
| 2025 年 11 月 | Claude Opus 4.5 / GPT-5.2 | Agent 遵循复杂指令的可靠性大幅提高 |
| 2026 年初 | **StrongDM "Dark Factory"** | 非交互式编程模式：仓库中只有 Markdown 规范，零人工代码，代码不由人写也不由人审查 |

**关键信号**：Microsoft、AWS、GitHub 三大平台在同一时期独立推出了 SDD 工具链，说明业界已形成共识——**"规范是新的代码"（Specs are the new code）**。开发者的角色从"写实现细节"转向"表达意图"——设计规范、引导 Agent、验证结果。

### 1.4 SDD 与软件工程的关系

一个常见的问题是：**"SDD 是软件工程应用在 AI Coding 的一种变式吗？"**

更准确的说法是：**SDD 不是软件工程的"变式"，而是软件工程中"规范先行"这一古老原则在 AI 时代的回归和强化。**

**回归**——传统软件工程一直强调"先想清楚再动手"。瀑布模型有需求规格说明书（SRS），RUP 有用例驱动，BDD 有 Gherkin 场景，DDD 有通用语言和限界上下文——每个时代的方法论都在回答同一个问题：**怎么在写代码之前确认大家对"要做什么"理解一致？** SDD 是这个传统的延续，不是新发明。

**强化**——区别在于执行层面：

| 时代 | 规范的角色 | 规范的命运 |
|------|-----------|-----------|
| 传统软件工程 | 给人看的参考文档 | 代码写到一半规范就过时了，没人维护 |
| **AI Coding 时代的 SDD** | **给 AI 的"唯一输入源"** | 规范不更新 = AI 产出错误代码，迫使规范必须保持鲜活 |

传统开发中，规范是"最佳实践"——违反的代价是沟通成本和返工，但人可以靠脑内上下文补救。AI 开发中，规范是"硬约束"——因为 AI 没有你的脑内上下文，规范是它理解项目的**唯一入口**。**规范中的遗漏 = AI 产出中的缺陷，这个因果关系比传统开发直接得多。**

本项目的实际经历印证了这一点：`auth.py` 中 JWT 解码被标记为 `# TODO` 占位，传统场景下可能口头约定"后面补"，AI 也不会主动发现——直到测试跑不过才暴露。如果 JWT 解码细节在技术方案中就写清楚（认证鉴权章节），AI 在实现 `auth.py` 时就能一次性做对。

**一句话概括**：SDD 把软件工程中"规范先行"的可选项，变成了 AI Coding 下的必选项。

## 二、我们的 SDD 实践全景

### 2.1 先看清边界：哪些是 SDD，哪些不是

本项目从两份规范文档出发，最终交付了一套可运行的系统。但**并非整个过程都是 SDD**。按 SDD 的标准定义（Specify → Plan → Tasks → Implement），实际过程可以分为三段：

```
┌─────────────────────────────────────────┐
│        🎯 SDD（规范驱动开发）本体          │
│  Specify → Plan → Tasks → Implement     │
│                                         │
│  起点：两份规范文档                        │
│  终点：全部代码写完（50+ 文件就绪）          │
│  特征：规范是"输入"，代码是"输出"            │
│  每步都对照文档验证                         │
└─────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│      🔧 AI 辅助验证与修复（非 SDD）         │
│                                         │
│  测试脚本编写 → 运行 → 排查 → 根因分析 → 修复  │
│  特征：规范的角色从"输入"变为"仲裁者"         │
│  代码写完了，现在验证它是否兑现了规范          │
└─────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│      🔗 跨服务集成与前端（非 SDD）          │
│                                         │
│  内部接口对接 → 安全设计 → 开发指南 → 前端浮窗  │
│  特征：以 shop-service 为上游的独立开发活动    │
│  有自己的需求和设计，遵循常规对接流程          │
└─────────────────────────────────────────┘
```

**为什么要划这条线？** 因为在 AI 辅助开发中，"写代码"和"修代码"用的是同一个工具（LLM），容易让人产生"都是一个流程"的错觉。但实际上：

- **SDD 阶段**：规范驱动代码。LLM 的行为是"文档说什么，我就做什么"。人类的主要工作是审核规范文档和任务清单。
- **修复阶段**：测试驱动修复。LLM 的行为是"日志报什么错，我就查什么根因"。规范的角色变成了判断"正确的行为应该是什么"的仲裁者。
- **集成阶段**：需求驱动对接。这是独立的开发活动，有自己的输入（AI 服务的 API 约定）和输出（全链路可用的系统）。

> **本文档的标注约定**：后续各章节标题将标注 `[SDD]`、`[验证]`、`[修复]`、`[集成]`，方便读者识别当前内容属于哪个阶段。

### 2.2 项目时间线

```
SDD 本体                          验证 & 修复                集成 & 交付
   │                                  │                         │
   ▼                                  ▼                         ▼
┌─────────┐  ┌──────────────────┐  ┌──────────────┐  ┌──────────────┐

```

### 2.3 文档演进史

规范文档本身不是一次写成的。本项目两份核心文档经历了多轮迭代：

**需求设计文档**（最终版 v5.3）：
- v1-v3：搭建骨架，确定章节结构，填充核心业务场景
- v4：补充状态机定义（第十章）、完善异常码体系（第十四章附录）
- v5.x：细化缓存策略、内部接口定义、添加 In/Out Scope 表格

**技术设计方案**（最终版 v2.0）：
- v1.0：初版含伪代码，各技术主题分散
- v2.0：**全文重构**——删除全部伪代码，定位调整为"设计决策与约定"；合并分散的技术主题；新增 ADR（架构决策记录）和实施检查清单

> **关键洞察**：技术方案 v1→v2 的重构体现了一个重要原则——**技术方案应该回答"我们选择了什么、为什么"，而不是"代码怎么写"**。伪代码会过时，决策记录不会。

### 2.4 本指南的阅读方式

本指南以项目实践的时间线为主线，每个阶段先讲述**"我们实际做了什么"**（案例叙事），再提炼**"为什么这样做"**（方法论抽象）。如果你只关心方法论，可以直接跳到第十章；如果你想完整了解一次 SDD 实践长什么样，建议从头到尾阅读。

## 三、起点：两份规范文档 `[SDD: Specify]`

### 我们做了什么

在写任何一行代码之前，我们手里已经有了两份文档：

1. **[需求设计文档](需求设计文档.md) v5.3** — 约 1330 行，覆盖 14 个章节
2. **[技术设计方案](技术设计方案.md) v2.0** — 约 606 行，覆盖 12 个章节

### 需求文档的核心内容

需求文档回答了"这个系统要做什么"：

- **项目背景**（第二章）：轻量级企业电商平台，覆盖用户→商品→订单→支付→物流→售后→AI 客服的完整交易闭环
- **范围边界**（第三章）：一张 In/Out Scope 表格，明确本期做用户注册/JWT/购物车/下单/支付/物流/售后/AI客服，不做什么手机验证码/SKU多规格/K8s/监控告警
- **用户角色**（第四章）：普通用户、管理员、AI 客服、定时任务调度器，各自的核心目标和权限边界
- **业务场景**（第五章）：C 端 18 个场景 + B 端 7 个场景，每个场景有编号、名称、触发条件、结果、优先级
- **功能设计**（第六章）：每个功能的"使用角色→前置条件→主处理逻辑→关键规则→异常说明"
- **业务流程图**（第八章）：PlantUML 流程图 + 文字步骤说明，覆盖下单、物流售后、AI 客服、商品管理、超时取消 5 条流程
- **状态机**（第十章）：所有实体的状态枚举、合法流转路径、**不可流转路径**（如 paid → cancelled 不可执行）
- **功能点清单**（第十一章）：C 端 25 个 + B 端 8 个 + 系统级 2 个功能点，每个有编号、输入、输出、关键规则、优先级
- **用户故事 + 验收规则**（第十二章）：每个用户故事对应 3-4 条可验证的验收规则
- **业务规则**（第十三章）：C 端 14 条 + B 端 6 条 + 并发控制 5 条 + 缓存策略 5 条 + 系统治理 7 条 + 容器化 5 条

### 技术方案的核心内容

技术方案回答了"这个系统要怎么做"：

- **技术选型**（第二章）：Python 3.10.10 + FastAPI + PostgreSQL 16 + pgvector + Redis + APScheduler + Nginx + Docker Compose
- **目录结构约定**（第二章）：`shop-service/` 下的 apps/domain/infrastructure/middleware/common 五层结构
- **事务管理约定**（第三章）：每种操作（下单/支付/取消/定时取消）的事务策略
- **中间件设计**（第四章）：RequestIDMiddleware、RequestLogMiddleware、全局异常处理器的设计决策
- **认证鉴权**（第五章）：JWT 字段定义、角色权限矩阵、内部接口 Token 认证
- **缓存方案**（第九章）：Cache-Aside 模式、3 个缓存 Key、失效矩阵、降级策略、并发竞态决策
- **架构决策记录 ADR**（第十章）：9 条关键决策及理由
- **实施检查清单**（第十一章）：数据库/状态机/定时任务/日志/内部接口/文档/缓存 7 个维度的 checklist

### 这个阶段的方法论

**① 两份文档的分工要清晰。** 需求文档写"用户下单时，FOR UPDATE 锁定库存，同一事务内扣库存→创订单→清购物车"，技术方案写"使用 psycopg2 手动事务，`conn.autocommit=False`，finally 中恢复 `autocommit=True` 并归还连接"。需求文档不应出现技术名词，技术方案不应重复业务规则。

**② In/Out Scope 表是最有价值的投入。** 花 30 分钟明确"不做什么"，可以省下后续无数讨论。本项目的 In/Out Scope 表在开发过程中被反复引用至少 10 次——"这个功能做不做？看范围表。"

**③ 状态机值得花时间画。** 订单的 pending→paid/cancelled 流转规则、不可流转路径（paid→cancelled 不允许）——这些信息如果只靠口述几乎一定会出 Bug。状态机的 PlantUML 图可以用 VS Code 插件实时渲染，开发时对照实现。

## 四、Step 1：从规范到任务 — LLM 如何规划开发 `[SDD: Plan & Tasks]`

### 我们做了什么

这一节是整个 SDD 流程中**最关键的一步**——拿着两份规范文档，如何变成可执行的开发任务清单？

实际过程：将需求文档和技术方案交给 LLM，指令是 **"请参照这两份文档规划我们开发任务"**。LLM 产出了一份分层任务清单，随后获得人工审核批准后开始执行。

### LLM 的任务规划逻辑

LLM 读了两份文档后，做了以下推演：

**① 从需求文档提取"要做什么"的完整列表。**

需求文档第十一章（功能点清单）有 35 个编号化功能点（FC1-FC25 + FB1-FB8 + FS1-FS2），每个标注了 P0/P1 优先级。LLM 过滤出全部 P0 功能点，这就是任务清单的"素材池"。

**② 从技术方案获取"代码放在哪"的分层约定。**

技术方案 2.3 节约定了五层目录结构：`apps/`（路由）、`domain/`（业务逻辑）、`infrastructure/`（基础设施）、`middleware/`（中间件）、`common/`（公共模块）。LLM 据此将功能点映射到具体的文件路径。

**③ 识别模块间的依赖关系，确定"什么必须先做"。**

这是最关键的一步。LLM 分析了模块间的依赖链：

```
common/exceptions.py     ← 被所有模块引用，必须最先做
infrastructure/database.py ← 被所有业务模块引用，必须在业务层之前
infrastructure/redis_client.py ← 被 product/category 模块引用
domain/cart/*            ← 独立模块，但 order 依赖它
domain/order/*           ← 下单后清空购物车，必须在 cart 之后
apps/internal/*          ← 调用 domain 层各 service，必须在业务层之后
main.py                  ← 组装所有路由，必须最后
```

**④ 将任务按依赖从底向上排列。**

最终产出的任务分层：

```
A 层 — 基础设施（无业务依赖，最先做）
  A1. 项目骨架      — 目录结构 + Dockerfile + requirements.txt + .env
  A2. 中间件         — RequestID + RequestLog + 异常处理
  A3. 公共模块       — Logger + 异常树 + Context
  A4. 基础设施       — DB 连接池 + Redis 客户端 + Scheduler

B 层 — 业务模块（依赖 A 层，模块间有部分依赖）
  B1. 用户模块      — 无业务依赖，可最先开始
  B2. 分类模块      — 无业务依赖，可与 B1 并行
  B3. 商品模块      — 依赖分类表，但可独立开发
  B4. 购物车模块    — 依赖商品和用户
  B5. 订单模块      — 依赖购物车（下单后清空）
  B6. 物流模块      — 依赖订单
  B7. 售后模块      — 依赖订单
  B8. 内部接口      — 依赖所有 domain 模块

C 层 — 集成与入口
  C1. 主入口        — main.py + lifespan + 路由挂载

D 层 — 部署配置
  D1. Nginx 配置
  D2. 数据库初始化   — init.sql
  D3. Docker Compose
```

**⑤ 标注每个任务的完成标准。**

任务清单中的每个任务都隐含了一个验证标准：该模块的 API 可被调用、业务逻辑符合需求文档描述、错误场景返回正确的异常码。

### 实际执行中的并行策略

有了依赖分析，LLM 在实施时做了并行调度：

- **第一批（并行）**：A1 骨架 + D1 Nginx + D2 init.sql（三者互不依赖）
- **第二批（并行）**：A2 中间件 + A3 公共层 + A4 基础设施（同属基础设施层）
- **第三批（并行）**：B1 用户 + B2 分类 + B3 商品 + B4 购物车（四个模块无相互依赖）
- **第四批（独立）**：B5 订单（依赖购物车，必须 B4 完成后才能开始）
- **第五批（并行）**：B6 物流 + B7 售后 + B9 定时任务
- **第六批（并行）**：B8 内部接口 + B10 主入口

最终在 **一个对话会话** 内完成了全部 17 个任务，创建约 50+ 个文件。

### 这个阶段的方法论

**① 规范文档的质量直接决定任务规划的质量。** LLM 能正确排序依赖，是因为需求文档的功能点清单有编号、技术方案的目录结构有约定、模块间的关系在业务流程图中被表达。如果文档模糊，任务规划也会模糊。

**② 人工审核任务清单是必须的。** LLM 产出的任务清单是"建议"，不是"命令"。本项目中，用户在 LLM 列出任务后进行了审核和确认，然后才开始执行。**SDD 的 "Specify → Plan → Tasks → Implement" 中，每个箭头前都应该有一个人工确认点。**

**③ 依赖分析是任务排序的核心算法。** 不是随机拆任务，不是按字母排序，不是按"哪个简单先做哪个"。唯一的排序依据是：**什么必须先存在，什么才能后存在**。这和 Makefile 的依赖解析、Docker 镜像的分层构建是同一个逻辑。

## 五、Step 2：从底向上构建 `[SDD: Implement]`

### 我们做了什么

拿到任务清单后，按照"不能跳跃依赖"的原则逐层实现。整个过程遵循一个固定的节奏：

```
选择目标模块 → 读取需求文档对应章节 → 读取已有代码了解风格 →
编写 service.py（业务逻辑）→ 编写 router.py（API 路由）→ 更新 __init__.py →
标记任务完成 → 进入下一层
```

### 骨架层：给代码一个"家"

第一步不是写业务代码，而是创建目录结构和配置文件：

- 25 个目录，每个放 `__init__.py`（先占位，后续覆盖）
- `Dockerfile`、`requirements.txt`、`.env`、`nginx.conf`、`docker-compose.yml`、`init.sql`

这些文件在后续开发中会被反复修改（如端口号调整、依赖增删），但骨架必须先搭好——不然代码没地方放。

### 公共层：被所有人依赖的先做

`common/` 目录下的三个文件是整个项目的"地基"：

| 文件 | 内容 | 被谁依赖 |
|------|------|---------|
| `exceptions.py` | 7 个子类的统一异常树 | 全部业务模块 |
| `context.py` | `ContextVar` 存储 `request_id` | 中间件 + Logger |
| `logger.py` | `RequestIDFilter` + `setup_logger()` | main.py 启动时初始化 |

这三个文件必须先完成，因为后续所有代码都会 `from common.exceptions import BusinessError` 或 `from common.logger import logger`。

### 中间件层：请求的"入关管道"

`middleware/` 下的三个文件处理每个 HTTP 请求的通用逻辑：

- `request_id.py`：从 Header 读取或生成 request_id → 存入 ContextVar → 回写响应头
- `request_log.py`：记录方法、路径、状态码、耗时
- `exception_handler.py`：全局捕获 `ShopException` → 映射为统一 JSON `{code, message, request_id}`

**关键设计点**：中间件不处理业务逻辑，只处理"每个请求都需要做的事"。业务逻辑的异常由各 service 抛出，中间件的异常处理器统一收口。

### 基础设施层：业务层的"插座"

`infrastructure/` 下的三个文件提供业务层依赖的外部资源：

- `database.py`：`ThreadedConnectionPool`（min=2, max=10），提供 `get_connection()` / `release_connection()`
- `redis_client.py`：Cache-Aside 模式的封装，`get_cache()` / `set_cache()` / `delete_cache()`
- `scheduler.py`：APScheduler `BackgroundScheduler`，后续为定时任务注册 job

### 业务层：一个模块一个模块地造

每个业务模块的实现模式是一致的——对照需求文档的对应章节：

**以订单模块为例：**

| 实现步骤 | 对照的文档章节 |
|---------|--------------|
| `create_order()` | 需求文档 §6.2.6：FOR UPDATE 锁库存 → 校验 → 扣减 → 创建订单+明细快照 → 清空购物车 → COMMIT |
| `pay_order()` | 需求文档 §6.2.7：FOR UPDATE 锁订单 → 校验 status=pending → 更新为 paid → 写支付记录 → COMMIT |
| `cancel_order()` | 需求文档 §6.2.8：FOR UPDATE 锁订单 → 校验 status=pending → 更新为 cancelled → 回滚库存 → COMMIT |
| 状态校验 | 需求文档 §10.2：只允许 pending→paid 和 pending→cancelled，paid→cancelled 拒绝 |
| `cancel_timeout_orders()` | 需求文档 §6.2.8：每 5 分钟扫描，超时 30 分钟的 pending 订单 → 逐条独立事务取消 |

**以商品模块为例：**

| 实现步骤 | 对照的文档章节 |
|---------|--------------|
| `get_product_detail()` | 技术方案 §9.2 Cache-Aside 模式：先查 Redis → 命中返回 → 未命中查 DB → 回写 Redis |
| `create_product()` / `update_product()` / `toggle_status()` | 技术方案 §9.4 缓存失效矩阵：每个 B 端写操作后删除受影响的 Redis key |
| `get_hot_products()` | 需求文档 §9.5：`hot:products:list`，前 5 条 on_sale 商品，TTL 5 分钟 |

### 入口层：最后的组装

`main.py` 是最后创建的文件——因为只有所有路由和中间件都就绪了，才能确定装配逻辑：

```python
# lifespan: 初始化 DB 池 → Redis → Scheduler → 注册定时任务
# middleware: RequestID → RequestLog → CORS
# routers: 11 个路由模块挂载
# /health: 健康检查
```

main.py 本身几乎没有业务逻辑——它只是一个"装配文件"，把所有模块连接起来。

### 这个阶段的方法论

**① "先读文档再写代码"是一个可执行的流程，不是口号。** 每实现一个模块，打开需求文档的对应章节，确认前置条件→主处理逻辑→关键规则→异常说明，然后才开始写。这和照着菜谱做菜是同一个道理——不先看菜谱就下锅，容易放错调料。

**② 同一层的模块可以并行，跨层不能跳跃。** 商品模块和分类模块可以同时写（互不依赖），但商品模块不能在 infrastructure 没建好之前写（它依赖 DB 和 Redis）。

**③ `__init__.py` 的导出是模块的"公共 API"。** 每个模块的 `__init__.py` 决定了对外的接口。如果一个函数没有在 `__init__.py` 中导出，外部模块就不应该调用它——这是一种 Python 风格的封装。

### 测试用例的来源

| 测试步骤 | 断言 | 来源 |

### 测试脚本的另一个角色

除了验收，测试脚本还充当了 **"可运行的需求文档"**。对于新加入的开发者和 AI Agent，跑一遍 `python test_shop.py` 比读半天文档更直观。测试的输出 = 系统行为的实际体现。

### 这个阶段的方法论

**② 测试脚本是规范的"活体"。** 文档会过时，测试不会——因为测试不通过就是红灯。始终让测试通过 = 始终让代码符合规范。

## 七、Step 4：Bug 驱动的收敛 `[修复阶段]`

### 我们做了什么

第一轮测试跑了 26/37 通过，11 项失败。本节回顾完整的排查和修复过程。

### 第一层：日志定位

所有容器通过 Docker json-file driver 输出日志。`docker logs shop-service` 直接查看，关键错误一目了然：

```
psycopg2.ProgrammingError: set_session cannot be used inside a transaction
POST /c-endpoint/orders HTTP/1.1" 500
```

11 个失败中有约 9 个的 URL 都是 `/orders/None/pay`、`/orders/None/cancel` 等——订单创建返回了 500，`ORDER_ID` 为 `None`，级联导致所有依赖订单的测试产生无意义的 422。

### 第二层：根因收敛

不是 11 个独立 Bug，而是 **2 个根因**：

**根因 1：`create_order` 的事务管理时序错误。**

`order_service.py` 中 `create_order` 的执行流程：

```
1. conn.autocommit = False        ← 进入手动事务模式
2. INSERT orders RETURNING id     ← 获取 order_id
3. INSERT order_items             ← 写入明细
4. DELETE cart_items              ← 清空购物车
5. conn.commit()                  ← 提交事务，事务结束
6. SELECT ... FROM orders         ← 💥 在 autocommit=False 下开启了新的隐式事务！
7. SELECT ... FROM order_items    ← 同上，仍处于隐式事务中
8. return order
9. conn.autocommit = True         ← 💥 崩溃！不能在有活跃事务时切换 autocommit
```

第 5 步 commit 后，代码又执行了 SELECT。在 psycopg2 中，`autocommit=False` 时任何 SQL 都会自动开启新事务。第 6、7 步的 SELECT 开启了新的隐式事务但没有提交，第 9 步的 `autocommit = True` 被 psycopg2 拒绝。

**修复**：让 INSERT 全部使用 `RETURNING` 子句，在 commit 之前收集完所有数据。修改后 commit 之后零 SQL，finally 安全执行。

**根因 2：JWT 认证是 TODO 桩代码。**

`auth.py` 中的 `get_current_user` 和 `get_current_admin` 的实现是：

```python
def get_current_admin(authorization: str = Header(None)) -> dict:
    if not authorization:
        raise AuthenticationError("未登录或 Token 过期")
    # TODO: 实际 JWT 校验逻辑
    return {"id": 1, "username": "admin", "role": "admin"}  # ← 永远返回 admin
```

JWT 在 `login()` 中已经正确生成（payload 含 `user_id`、`email`、`role`），但认证层从未解码验证。任何带 Authorization 头的请求都能通过，`get_current_admin` 更不会校验角色是否为 admin。

**修复**：实现 `get_current_user`（解码 JWT，返回 `{user_id, email, role}`）和 `get_current_admin`（调用 `get_current_user` 后校验 `role == "admin"`）。

### 第三层：还有 2 个"失败"？

修复后回归测试：35/37 通过，2 项"失败"。

深入分析发现，这 2 项不是 Bug：

| 测试项 | 实际行为 | 分析 |
|--------|---------|------|
| 重复注册 | API 返回 HTTP 400 + code 40001 | 需求文档 §14.4 规定 `ValidationError` → HTTP 400。测试断言写了 422，**是测试写错了，不是代码错了** |
| 权限隔离 | 普通用户访问 B 端返回 403 | JWT 修复后已正确拦截，测试断言更新后通过 |

### 这个阶段的方法论

**① 日志是第一线索。** `RequestIDMiddleware` 让每个请求的错误都带上了 `request_id`，可以通过 `docker logs` 一键追踪单次请求的完整日志链路。

**② 不要看到一个 Bug 修一个，先找共性根因。** 11 个失败 → 2 个根因 → 修复后全部消除。面对大量测试失败时，先问："这些失败有没有共同的上游依赖？"

**③ 对照文档验证"正确行为"而非凭直觉。** "重复注册应该返回 422"——文档说的是 `ValidationError` → HTTP 400。测试断言错了。**规范是判断"对错"的最终仲裁者。**

**④ 修复方案要治本不治标。** `autocommit` 崩溃的本质是"commit 之后不应该再有 SQL"。修复方案不是给 finally 加 try/except 容错（治标），而是让 INSERT 用 RETURNING 收集数据、commit 之后零 SQL（治本）。

### 这一段如何验证了 SDD 的价值

修复阶段虽然不是 SDD 本体，但它恰恰是 **SDD 价值最大的验证时刻**。三段修复分别对应了规范的三种用法：

| 修复项 | 怎么发现的 | 规范的角色 |
|--------|-----------|-----------|
| `autocommit` 时序 bug | 对照技术方案 §3.3 事务管理约定，发现代码违背了设计 | **规范作为"正确行为的参照系"** |
| JWT 认证 TODO 桩代码 | 对照技术方案 §5 认证与鉴权章节，发现代码遗漏了实现 | **规范作为"完整性检查清单"** |
| "重复注册应该返回 422" | 对照需求文档 §14.4 异常码表，发现是测试断言写错了不是代码错了 | **规范作为"争议的最终仲裁者"** |

**如果规范没写好，这三个 Bug 根本无法精确定位。** 没有事务约定文档，`autocommit` 崩溃只能靠猜。没有认证鉴权设计，JWT 缺失可能被视为"功能就是这样"。没有异常码表，"422 vs 400" 的争论永远没有答案。

> **SDD 的核心价值不在"写代码快"，而在"定位问题准"。规范文档的价值在 Bug 修复时达到最大化——它让你知道"正确的行为应该是什么"，而不是凭记忆和感觉猜测。

## 八、Step 5：跨服务对接 `[集成阶段]`

### 我们做了什么

shop-service 开发完成后，需要与 AI 客服服务对接。这涉及三个层面的工作。

### 8.1 确认内部接口就绪

对照技术方案 8.1 节，验证 7 个内部接口全部正确实现：

| 端点 | 功能 | 认证 |
|------|------|------|
| `GET /internal/orders` | 用户订单列表 | `X-Internal-Token` |
| `GET /internal/orders/{id}` | 订单详情 | `X-Internal-Token` |
| `GET /internal/logistics` | 用户物流 | `X-Internal-Token` |
| `GET /internal/after-sales` | 用户售后 | `X-Internal-Token` |
| `GET /internal/products/search` | 商品搜索 | `X-Internal-Token` |
| `GET /internal/products/{id}` | 商品详情 | `X-Internal-Token` |
| `GET /internal/users/{id}` | 用户信息 | `X-Internal-Token` |

**发现并修复了一个关键 Bug**：内部接口的 `verify_internal_token` 调用方式不是 FastAPI 的 `Depends` 依赖注入，而是普通函数调用中传入 `Header(...)`。`Header(...)` 在普通函数调用中创建的是 `FieldInfo` 对象而非解析后的值，导致 Token 字符串比较永远失败——所有内部接口一律返回 401。

修复方式：将 7 处调用改为 `_token: bool = Depends(verify_internal_token)`，让 FastAPI 的依赖注入系统正确解析 Header。

### 8.2 安全设计：user_id 不能由前端传递

AI 服务查询用户订单时，`user_id` 从哪里来？**不能从前端请求参数中获取**——任何用户都可以篡改参数查看他人数据。

正确方案：AI 服务共享 `JWT_SECRET`，自己解码用户 JWT 提取 `user_id`，然后调用 shop 内部接口：

```
前端 → AI Service（解码 JWT → user_id=5）→ shop-service（/internal/orders?user_id=5）
```

### 8.3 输出对接指南

为 AI 服务团队输出了 [AI服务对接开发指南](AI服务对接开发指南.md)，包含：
- 调用架构（Docker 内网直连，不经 Nginx）
- 可复用代码位置（JWT 解码参考 `shop-service/apps/common/auth.py`）
- `ShopInternalClient` 封装类（7 个接口的完整调用代码）
- 环境变量清单和安全注意事项

### 8.4 前端集成

开发了全局 AI 客服浮窗组件（`ChatPopup.jsx`），修复了从前端到 AI 服务的全链路路由：

```
开发模式：Vite (5173) → /api/ai/chat → rewrite → localhost:8000/api/chat
Docker 模式：用户 → Nginx (83) → /api/ai/ → rewrite → ai-service:8000/api/
```

### 这个阶段的方法论

**① 内部接口不应该暴露在公网。** Nginx 在 `/api/shop/` 之前增加了 `/api/shop/internal/` 拦截规则返回 403。AI 服务通过 Docker 内网 `http://shop-service:8004/internal/` 直连，不走 Nginx。

**② 服务间调用选择最简单的方案。** 对于"AI 查询实时数据后生成回复"这种同步请求-响应场景，HTTP 直连就足够了。不引入 gRPC（需要两边写 proto + stub）或消息队列（增加异步延迟和复杂度）。

**③ 为协作方写"对接指南"，不是扔一份接口文档就完事。** 对接指南应该包含：可复用的代码片段、环境变量清单、调用示例、安全注意事项。让协作方能"照着复制粘贴就能跑"。

## 九、关键决策回顾

本节以 ADR（Architecture Decision Record）风格回顾项目中的关键设计决策：

| 编号 | 决策 | 当时的选择 | 理由 | 事后验证 |
|------|------|-----------|------|---------|
| D1 | **任务拆解策略** | 自底向上，按依赖排序 | 基础设施不建好，业务代码写不了 | ✅ 一次性完成全部 17 个任务，零返工 |
| D2 | **技术方案定位** | 写设计决策，不写伪代码 | 伪代码会过时，决策不会 | ✅ v2.0 重构后文档一直保持有效 |
| D3 | **缓存并发竞态** | 单次删除，不引入延迟双删 | 单机部署窗口毫秒级，`FOR UPDATE` 兜底，TTL 自愈 | ✅ 未出现缓存不一致问题 |
| D4 | **Redis 不做持久化** | 不做 | 数据可从 DB 完全重建 | ✅ 节省资源，Docker 数据卷管理简单 |
| D5 | **内部 API 认证** | 固定 Token + .env 注入 | Demo 足够，生产建议 Vault 轮换 | ✅ 部署简单，但需要注意 Token 同步 |
| D6 | **JWT SECRET 共享** | AI 服务与 shop-service 共享 | 避免多一跳网络调用去验证 Token | ✅ AI 服务可独立解码用户身份 |
| D7 | **AI 服务调用方式** | 直接 HTTP（httpx） | 同步请求-响应，零额外依赖 | ✅ 调试方便，curl 即可测试 |
| D8 | **超时阈值 30min + 扫描间隔 5min** | 30min/5min | Demo 可快速验证 | ✅ 定时任务正常运行 |
| D9 | **单机部署** | 不考虑 K8s/多副本 | 教学 Demo 定位 | ✅ Docker Compose 一键启动 |

## 十、方法论提炼：可复用的 SDD 模式

从本项目的实践中提炼出 5 条可复用的原则和 1 个模板。

### 原则 1：规范是"可验证"的

好的规范不是"系统要有良好的用户体验"，而是"注册时邮箱不可重复，密码 ≥ 6 位，登录成功返回有效期 24h 的 JWT Token"。后者可以在测试中逐条验证。

**检验方法**：读完一条规范描述后问自己——"我能写一个测试用例来验证这条吗？"如果不能，规范就不够精确。

### 原则 2：规范先于代码，但规范可以随代码修正

SDD 不意味着文档"写完就冻结"。本项目需求文档迭代了 5 个版本，技术方案迭代了 2 个版本。关键是变更要有记录，不能"默默改掉"。**在动手前把大头定下来，但不排斥实现过程中回头补充发现遗漏。**

### 原则 3：范围边界是最高优先级的信息

需求文档第三章（In/Out Scope）每一条"不做"都节省了无数讨论。**把不做什么写清楚，比把做什么写清楚更难，也更有价值。** 开发过程中遇到的所有"这个做不做"的问题，答案都应该能在范围表中找到。

### 原则 4：分层实现，不能跳跃

```
骨架 → 公共层 → 基础设施 → 领域服务 → API 路由 → 入口组装 → 测试
```

跳层会导致返工。如果先写 API 路由再写基础设施，写完了会发现 DB 连接池还没建好，全部代码要改。

### 原则 5：测试必须可运行

测试脚本是"规范的可执行版本"。如果测试脚本因为前置条件缺失而跑不起来，它就失去了验证能力。本项目在编写测试脚本的同时准备了预置种子数据（管理员账号、9 个分类、10 个商品），确保 `docker compose up -d && python test_shop.py` 一条命令就能完整验证。

### 可复用模板：三层文档结构

本项目演进出了一套三层文档结构，可用于任何 SDD 项目：

```
第一层：需求设计文档  ─── 回答 "要做什么"
   ├─ 项目背景、范围边界、用户角色
   ├─ 业务场景、功能设计、流程图
   ├─ 状态机、功能点清单、用户故事、验收规则
   └─ 面向：产品、开发、测试、AI Agent

第二层：技术设计方案  ─── 回答 "怎么做"
   ├─ 技术选型、服务拓扑、目录结构
   ├─ 数据库设计、中间件设计、认证鉴权
   ├─ 缓存方案、日志方案、ADR、实施检查清单
   └─ 面向：开发、架构评审、AI Agent

第三层：开发指南       ─── 回答 "怎么上手 / 怎么对接"
   ├─ 可复用代码位置、环境变量清单
   ├─ 接口速查表、调用示例、安全注意事项
   └─ 面向：协作方、新加入的开发者
```

三层的关系是递进的：
- 需求文档 §6.2.12 说"AI 客服能查询用户订单"
- 技术方案 §8.1 说"通过 `/internal/orders?user_id=...` 接口 + `X-Internal-Token` 认证"
- 开发指南说"`ShopInternalClient.get_orders(user_id)`，JWT 解码参考 `shop-service/apps/common/auth.py`"

## 十一、常见陷阱与避坑指南

以下陷阱全部来自本项目的实际经历。

### 陷阱 1：技术方案写成伪代码集合

**现象**：技术方案 v1.0 贴了大量伪代码，改代码后文档没跟着更新，文档信用破产。

**避坑**：技术方案只写设计决策，不写伪代码。写"用 Cache-Aside 模式，读未命中回写，写操作删缓存"，不写 `if redis.get(key) else db.query()`。

### 陷阱 2：commit 之后又执行了 SQL

**现象**：`conn.commit()` 之后执行了 SELECT 查询，在 `autocommit=False` 下开启了新的隐式事务，finally 中 `autocommit = True` 崩溃。

**避坑**：所有需要的数据用 `RETURNING` 子句在事务内收集完毕，commit 之后执行零 SQL。如果确实需要在事务外读数据，先恢复 `autocommit = True`。

### 陷阱 3：认证逻辑标 TODO 后忘记回来填

**现象**：`auth.py` 中的 JWT 解码留了 `# TODO: 实际 JWT 校验逻辑`，之后忘了补。导致所有带任意 Authorization 头的请求都能冒充 admin。

**避坑**：实施检查清单中有"认证鉴权"专项，提测前逐条核验。TODO 标记应该在代码审查阶段被识别为阻塞项。

### 陷阱 4：测试断言凭感觉写，不查文档

**现象**：测试写了 `assert response.status_code == 422`，但文档定义是 `ValidationError → HTTP 400`。断言错了，不是代码错了。

**避坑**：测试断言的期望值从需求文档的异常码定义表（§14.4）来，不从感觉来。

### 陷阱 5：Bug 修复不分根因，逐个打补丁

**现象**：11 个测试失败就逐个修 11 个地方，结果是修了症状没修根因，同样的 Bug 改天换个地方又出现。

**避坑**：先看全局 → 找共性 → 分析根因 → 确认方案 → 再动手。本项目的 11 项失败根因只有 2 个。**如果你的 fix 列表比根因列表长，你在修症状。**

## 十三、小结

本项目实践了一次完整的 SDD 闭环和后续的 AI 辅助迭代：

- **SDD 本体**（第三至五章）：两份规范文档 → LLM 任务规划 → 分层实现 → 50+ 文件就绪
- **验证与修复**（第六至七章）：测试脚本 → Bug 定位 → 根因分析 → 修复回归——规范从"输入"变成了"仲裁者"
- **集成与交付**（第八章）：跨服务对接、前端浮窗、全链路贯通

核心体会：

> **规范决定了代码的上限，也决定了修复的下限。** 规范把状态流转规则写清楚了，代码就不会犯状态机错误；规范把异常码表写清楚了，测试就不会凭感觉写断言；规范把事务约定写清楚了，Bug 排查时就有客观的参照系。**规范是代码的"锚"——SDD 阶段它驱动你写对代码，修复阶段它告诉你什么是"对"。**

三条最重要的经验：

1. **先把"要做什么"和"怎么做"写成文档，再让 AI 动手。** 文档不是负担，是后续每一步都变简单的投资。没有文档的 AI Coding 是"凭感觉编程"——快在开头，慢在修复。
2. **SDD 的回报在 Bug 修复时达到峰值。** 写代码时省的时间，修 Bug 时会加倍还回去。规范和代码之间不是"文档 vs 代码"的对立——规范是让代码变得可维护的杠杆。
3. **测试是规范的"活体"。** 文档可能读完就忘，但测试红了你一定会修。验收规则 → 测试断言，这条翻译链是 SDD 质量保证的最后一道防线。
