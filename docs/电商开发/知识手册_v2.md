## 第1章 前置知识

> 三个会在后续章节反复出现的工具概念。本章只需看懂概念和用法，不需要理解它们在本项目中的具体作用——那个在遇到时会回顾。

### 1.1 ContextVar：协程安全的"便签"

**① 问题**：假设你想在请求进来时存一个 `request_id`，然后在后续处理中（日志、异常响应）读取它。如果你用普通全局变量：

```python
current_request_id = "-"

# 请求 A 进来
current_request_id = "abc123"
# ... 协程切换到请求 B ...
current_request_id = "def456"   # ← 覆盖了请求 A 的值！
# ... 切回请求 A ...
print(current_request_id)       # "def456" —— 错了，应该是 "abc123"
```

FastAPI 是异步框架，多个协程可能交错执行。普通全局变量会被互相覆盖。

**② 概念**：`ContextVar` 是 Python 3.7 引入的**协程局部变量**。每个协程有自己独立的"副本"——协程 A 写入的值，协程 B 读不到。类比：每个协程有一张独立的便签纸，彼此不干扰。

**③ 用法**：

```python
from contextvars import ContextVar

# 定义一个 ContextVar，默认值 "-"
request_id_var: ContextVar[str] = ContextVar("request_id", default="-")

# 写入（当前协程）
request_id_var.set("abc12345")

# 读取（当前协程）
current = request_id_var.get("-")  # 未设置时返回 "-"
```

后续章节中，你会看到中间件用 `.set()` 写入 request_id，日志 Filter 用 `.get()` 读出——但它们的核心机制就是这三行。

### 1.2 统一响应格式

**① 问题**：一个系统有几十个接口。如果每个接口返回的 JSON 结构不一致，前端需要为每个接口写不同的解析逻辑，一旦出错前端无法统一处理。

**② 概念**：本项目所有接口（C 端、B 端、内部接口）统一返回：

```json
{"code": 0, "data": {...}, "message": "success"}
```

| 字段 | 含义 | 示例 |
|------|------|------|
| `code` | 0 = 成功，非 0 = 异常 | `0`, `42201` |
| `data` | 业务数据（对象或数组） | `{"id": 42, "status": "paid"}` |
| `message` | 可读的描述 | `"success"`, `"库存不足"` |

列表接口在 `data` 中额外包含 `page`、`size`、`total`。

**③ 异常码速查**（后续章节中看到错误码时回这里查）：

| code | 含义 | HTTP 状态码 |
|------|------|------------|
| 0 | 成功 | 200 |
| 40001 | 参数校验失败 | 400 |
| 40101 | 未登录或 Token 过期 | 401 |
| 40301 | 无操作权限 | 403 |
| 40401 | 资源不存在 | 404 |
| 42201 | 业务规则冲突（库存不足、状态不允许等） | 422 |
| 50001 | 数据库异常 | 500 |
| 50002 | 未预期的内部错误 | 500 |

### 1.3 Python logging 骨架

**① 问题**：项目中到处都在打印日志——中间件记录请求耗时、service 记录订单创建、scheduler 记录超时取消。如果每个地方都 `print()`，你无法控制格式、无法按级别过滤、无法自动附加 request_id。

**② 概念**：Python 的 `logging` 模块由四个角色协作：

```
Logger            "我要记录一条日志"  —— logger.info("订单创建成功")
    │
    ▼
Filter            "这条日志需要附加什么信息？"  —— 自动注入 request_id
    │
    ▼
Formatter         "日志长什么样？"  —— 时间 | 级别 | request_id | 位置 | 消息
    │
    ▼
Handler           "日志输出到哪？"  —— sys.stdout（Docker 自动收集）
```

- **Logger**：你代码里调用的对象，`logger.info(...)`、`logger.error(...)`
- **Filter**：在日志被输出前附加额外字段（如从 ContextVar 取出 request_id）
- **Formatter**：定义日志字符串的格式模板
- **Handler**：把格式化后的日志写到某个目标（stdout、文件、网络）

**③ 用法**（项目中你只需要知道这段配置干了什么）：

```python
# 创建 Formatter——定义格式模板
formatter = logging.Formatter(
    "%(asctime)s | %(levelname)-7s | %(request_id)s | %(name)s:%(lineno)d | %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
)

# 创建 Handler——输出到 stdout
handler = logging.StreamHandler(sys.stdout)
handler.setFormatter(formatter)
handler.addFilter(RequestIDFilter())   # ← Filter 在这里挂载

# 把 Handler 绑到根 Logger
root_logger = logging.getLogger()
root_logger.addHandler(handler)
```

本章你只需要记住这四者的关系图。后面遇到"日志 Filter 注入 request_id"时，你会明白它在四步中的位置。

## 第2章 第一个请求：下单

> 主线：追踪 `POST /c-endpoint/orders` 从 Nginx 到 FastAPI 再到数据库的完整旅程。

我们从一个具体场景出发——用户已登录，购物车里有商品，点击"结算"按钮：

```
POST /api/shop/c-endpoint/orders
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
Body: {"address": "广东省深圳市南山区"}
```

下面按这个请求经过的每一站依次展开。

---

### 2.1 第〇站：请求到达之前 —— FastAPI 是什么

在追踪请求之前，需要先理解两层"入口"：一是 Nginx（容器化部署中介绍），二是 FastAPI 本身。

**① FastAPI 是什么**：一个 Python 异步 Web 框架。它的核心工作只有一件事——**把 HTTP 请求映射到 Python 函数，把 Python 返回值映射为 HTTP 响应**。

**② 最简形态**：

```python
from fastapi import FastAPI

app = FastAPI(title="电商平台 - Shop Service", version="1.0.0")

@app.get("/health")
def health_check():
    return {"status": "ok"}   # dict 自动序列化为 JSON
```

`@app.get("/health")` 告诉 FastAPI：当 `GET /health` 请求到来时，调用 `health_check()` 函数。你不需要手动处理 HTTP 协议。

---

### 2.2 第一站：中间件链 —— RequestID 进门

请求进入了 FastAPI。但到路由函数之前，它先穿过一层**中间件链**。

**什么是中间件**：位于请求和路由函数之间的一层统一处理逻辑。所有请求——不管最终调哪个路由——都会经过中间件。

```
请求 → [中间件A] → [中间件B] → [路由函数] → [中间件B] → [中间件A] → 响应
```

本项目注册了三个中间件，执行顺序如下：

**第一个：RequestIDMiddleware** —— 给请求贴身份标签。

**① 问题**：系统同时处理几十个请求。出 bug 时，你在日志里看到一条"库存不足"，但不知道是哪个用户的哪个请求触发的。你需要每个请求有一个唯一 ID，贯穿它的所有日志。

**② 概念**：中间件在请求进入时读/生成一个 `request_id`，存入 `ContextVar`（回顾 §1.1），然后在响应返回时把它写进响应头。

```python
class RequestIDMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        # 读请求头，没有则生成 8 位 UUID
        request_id = request.headers.get("X-Request-ID") or uuid.uuid4().hex[:8]
        request_id_var.set(request_id)          # ← 回顾 §1.1：写入 ContextVar

        response = await call_next(request)     # ← 交给下一个中间件

        response.headers["X-Request-ID"] = request_id  # 回写响应头
        return response
```

> **关键细节**：`call_next(request)` 是核心调用——它把请求交给下一个中间件（或路由函数）。不调用它，请求就"卡"在这里了。`await` 之后的代码在路由函数执行完毕后运行——这就是为什么回写响应头写在 `call_next` 之后。

**第二个：RequestLogMiddleware** —— 记录每个请求的方法、路径、耗时。

```python
class RequestLogMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        start = time.time()
        response = await call_next(request)
        elapsed = (time.time() - start) * 1000  # 毫秒

        logger.info("%s %s → %d | %.2fms",
            request.method, request.url.path,
            response.status_code, elapsed,
        )
        return response
```

这条日志输出时，`RequestIDFilter`（§1.3 中的 Filter）会自动从 `ContextVar` 取出 request_id 注入——所以日志里包含了 request_id，即使代码里没有显式传它。

**第三个：CORSMiddleware** —— 允许浏览器跨域调用。这是 FastAPI 内置的，不做展开。

### 2.3 第二站：路由匹配 —— 哪个函数处理这个请求

中间件链走完后，FastAPI 根据 `POST /c-endpoint/orders` 查找匹配的路由函数。找到了：

```python
# 在 apps/c_endpoint/order/router.py 中
router = APIRouter(prefix="/c-endpoint/orders")

@router.post("")
def c_create_order(req: CreateOrderRequest, user: dict = Depends(get_current_user)):
    # ...
```

> **路由怎么组织**：`APIRouter` 把同模块的接口分组——订单相关路由在一个 Router 里，商品相关在另一个。`main.py` 用 `app.include_router(order_router)` 挂载。这部分做一遍就明白，不展开。

### 2.4 第三站：Depends —— 在函数执行前拦截

路由匹配后，FastAPI 发现 `c_create_order` 的参数有 `Depends(get_current_user)`。在调用函数之前，它先执行 `get_current_user()`，把返回值传给 `user` 参数。

这就是 **FastAPI 的依赖注入（Depends）**——把"提取当前用户"这个逻辑从路由函数中抽出来。

**① 问题**：几乎每个 C 端路由都需要知道"谁在操作"。如果每个函数都手写"从 Header 取 Token → 解析 JWT → 返回 user_id"，代码极冗余。

**② 概念**：`Depends` 把这段逻辑封装成一个函数，FastAPI 自动调用并传参。

```python
def get_current_user(authorization: str = Header(None)) -> dict:
    """从 Authorization 头提取 JWT，验证后返回用户信息"""
    if not authorization:
        raise AuthenticationError("未登录")

    scheme, token = authorization.split(" ", 1)  # "Bearer <token>"
    payload = jwt.decode(token, JWT_SECRET, algorithms=["HS256"])
    return {
        "user_id": payload["user_id"],
        "email": payload["email"],
        "role": payload["role"],
    }
```

**JWT 是什么**：无状态认证令牌。服务端登录时签发（包含 `user_id` + `role` + `exp` 过期时间），客户端后续每次请求都带这个 Token。服务端验证签名后就知道是谁——不需要查数据库。

```
eyJhbGciOiJIUzI1NiJ9 . eyJ1c2VyX2lkIjoxLCJyb2xlIjoidXNlciJ9 . HMAC签名
      Header              Payload (Base64，不是加密的)          Signature (防篡改)
```

登录时 bcrypt 验证密码、JWT 签发的细节此处不展开——它们属于"登录请求"的旅程，不在"下单请求"的路径上。

**③ 对本请求的影响**：`get_current_user()` 返回 `{"user_id": 7, "email": "user@x.com", "role": "user"}`，这个 dict 后续传给 `c_create_order` 的 `user` 参数，也传给所有需要 `Depends(get_current_user)` 的路由。

> **管理员权限**：B 端路由用 `Depends(get_current_admin)`——它在 `get_current_user` 基础上多校验一步 `role == "admin"`，否则 raise 403。内部接口用 `Depends(verify_internal_token)`——校验固定 Token 而非 JWT。这些在不同请求中各自用到时自然理解。

---

### 2.5 第四站：Pydantic —— 请求体校验

`get_current_user` 通过后，FastAPI 处理 `req: CreateOrderRequest`。这是一个 Pydantic Model：

```python
class CreateOrderRequest(BaseModel):
    address: str

    @field_validator("address")
    @classmethod
    def address_not_empty(cls, v: str) -> str:
        if not v or not v.strip():
            raise ValueError("收货地址不能为空")
        return v.strip()
```

**① 问题**：前端发来的 JSON 不可信任——`address` 可能是 `""`、可能没传。每个路由手写校验代码会极度冗余。

**② 概念**：Pydantic 是 Python 数据校验库。你声明一个 `BaseModel` 子类描述请求体结构，FastAPI 在路由函数执行前自动校验。类型不对、必填字段缺失、自定义规则不通过——都在到达函数之前返回 422 错误。

**③ 对本请求的影响**：如果 `{"address": ""}`，`c_create_order` 函数**根本不会执行**，直接返回校验错误。校验通过后，`req.address` 已经是 `"广东省深圳市南山区"`（已 strip）。

### 2.6 第五站：业务逻辑 —— 下单的事务与锁

认证和校验全部通过，`c_create_order` 函数体开始执行：

```python
@router.post("")
def c_create_order(req: CreateOrderRequest, user: dict = Depends(get_current_user)):
    order = create_order(user["user_id"], req.address)
    return {"code": 0, "data": order, "message": "success"}
```

路由函数只做三件事：**校验参数、调用 Service、格式化返回**。真正的业务逻辑在 `create_order()` 中。

---

#### 2.6.1 连接从哪来

`create_order()` 需要查数据库。它调用 `get_connection()` 获取一个数据库连接。但这个连接不是临时新建的——

```python
# 在 infrastructure/database.py 中
_pool = ThreadedConnectionPool(
    minconn=2,   # 最少保持 2 个空闲连接
    maxconn=10,  # 最多 10 个
    dsn="postgresql://user:1234@postgres:5432/agent",
)

def get_connection():
    return _pool.getconn()   # 从池中取一个已有连接
```

**为什么用连接池**：建立一次数据库连接需要 TCP 三次握手，约 1-3ms。如果每次请求都新建销毁，这个开销远超执行 SQL 本身。连接池预创建一批连接，用完归还，不关闭。2-10 个连接足够本项目单机部署的并发量。

---

#### 2.6.2 游标：执行 SQL 并读取结果

拿到连接后，需要创建**游标（Cursor）**来执行 SQL。

**什么是游标**：数据库返回一个结果集，游标是指向这个结果集的"指针"。每次 `fetchone()` 读一行并下移指针。类比 Java JDBC 的 `ResultSet`——`resultSet.next()` 移指针，`resultSet.getString("email")` 读值。

psycopg2 默认游标返回元组（`row[0]`、`row[3]` 这种方式不可读）。本项目指定 `RealDictCursor`，让 `fetchone()` 返回 dict 风格的对象，用字段名访问：

```python
cur = conn.cursor(cursor_factory=RealDictCursor)
cur.execute("SELECT id, email FROM shop.users WHERE id = %s", (1,))
row = cur.fetchone()
print(row["email"])   # 用字段名，非 row[1]
```

---

#### 2.6.3 普通查询：用 `get_cursor()` 简化

大多数查询（查用户、查商品列表）只需要一条 SELECT，执行完自动提交即可。项目封装了 `get_cursor()` 上下文管理器来简化这类操作：

```python
# with 块 = 一个自动事务
with get_cursor() as cur:
    cur.execute("SELECT * FROM shop.users WHERE id = %s", (1,))
    return dict(cur.fetchone())
# ← 退出 with：自动 commit（成功）或 rollback（异常），自动归还连接
```

你只需要写中间的 SQL，连接管理、事务控制、资源释放全部自动。代码中这个模式覆盖了约 80% 的数据库操作。

---

#### 2.6.4 手动事务：下单需要更强控制

但 `create_order()` 不能用 `get_cursor()`。因为下单包含五步——锁库存 → 校验 → 扣库存 → 创订单 → 清购物车——这些必须在**同一个事务**中完成（要么全做，要么全不做），而且中间不能被其他请求干扰。

**什么是事务**：一组 SQL 的原子执行单元。`BEGIN → 操作1 → 操作2 → 操作3 → COMMIT`，中间任何一步失败，`ROLLBACK` 全部回滚。

`get_cursor()` 的问题在于：你只能在 with 块结束时一次性 COMMIT。但下单中途可能需要"发现库存不足就立刻 ROLLBACK 并抛出异常"——你需要手动控制 COMMIT 时机。

```python
conn = get_connection()
conn.autocommit = False    # ← 关闭自动提交，从此你手动控制

try:
    cur = conn.cursor()
    # ... 五步操作 ...
    conn.commit()           # ← 全部成功，手动 COMMIT
except Exception:
    conn.rollback()         # ← 任何一步失败，全部回滚
    raise
finally:
    conn.autocommit = True  # 恢复，归还后不影响下次使用
    cur.close()
    release_connection(conn)
```

> **`autocommit` 是什么**：psycopg2 的**事务开关**。`autocommit=True`（默认）时，每条 SQL 执行后自动 COMMIT，每条 SQL 自成一个事务——你感觉不到"事务"的存在。`autocommit=False` 时，第一条 SQL 执行时 psycopg2 自动向数据库发送 `BEGIN`，之后所有的 SQL 都在同一个事务中，直到你显式调用 `commit()` 或 `rollback()`。代码中那句 `conn.autocommit = True` 的恢复很重要——归还连接池后，下一个使用者期望的是默认行为（自动提交）。

---

#### 2.6.5 FOR UPDATE：防止超卖

下单五步中最关键的一步是锁库存。假设某个商品库存只有 6 件，两个用户几乎同时下单——用户 A 买 5 件，用户 B 买 3 件。

`SELECT ... FOR UPDATE` 对查到的行加**排他锁**：

```python
# 这个查询对查出的每一行加锁
cur.execute(
    "SELECT id, stock FROM shop.products WHERE id IN (...) FOR UPDATE",
    product_ids,
)
```

时间线：

```
用户 A 的请求：SELECT ... FOR UPDATE → 锁住商品行（库存 6）
用户 B 的请求：SELECT ... FOR UPDATE → 同一行，等待 A 释放锁
用户 A 的请求：校验通过 → 扣 5 → COMMIT → 释放锁（库存变 1）
用户 B 的请求：拿到锁 → 看到库存 1 < 3 → ROLLBACK → "库存不足"
```

如果没有 FOR UPDATE，两个请求同时看到库存 6，都认为够，都扣——超卖 2 件。

> **"排他"和"行级"的含义**："排他"指同一行上同一种锁模式同时只能由一个事务持有——事务 B 的 `FOR UPDATE` 必须等事务 A COMMIT 或 ROLLBACK 后才能继续。"行级"指锁的粒度是行，不是整张表——事务 A 锁了商品 5 的行，事务 B 可以同时锁商品 8 的行，互不干扰。锁在 COMMIT 或 ROLLBACK 时自动释放，不需要手动 UNLOCK。

**项目中用到 FOR UPDATE 的场景**：

| 场景 | 锁定对象 | 目的 |
|------|---------|------|
| 创建订单 | `shop.products` 的对应行 | 防止超卖 |
| 支付 | `shop.orders` 的对应行 | 防止重复支付 |
| 取消订单 | `shop.orders` 的对应行 | 防止并发取消和支付冲突 |

---

#### 2.6.6 下单的全部五步

回到 `create_order()`，完整的事务内操作序列：

```
conn.autocommit = False
  ├── ① 查购物车（JOIN products 获取价格和库存）
  ├── ② FOR UPDATE 锁定商品行
  ├── ③ 逐条校验库存 → 不足则 ROLLBACK + raise BusinessError("库存不足")
  ├── ④ 逐条扣减库存 + INSERT orders + INSERT order_items + DELETE cart_items
  └── ⑤ COMMIT
```

下单成功，`create_order()` 返回订单对象。

---

### 2.7 第六站：中途出错 —— 异常处理器接管

假设第 ③ 步抛出了 `BusinessError("库存不足")`。路由函数中的代码从未执行到 `return`——异常向上传播，被**全局异常处理器**捕获：

```python
# 在 middleware/exception_handler.py 中注册
@app.exception_handler(ShopException)
async def handler(request, exc: ShopException):
    return JSONResponse(
        status_code=exc.http_status,
        content={
            "code": exc.code,
            "message": exc.message,
            "request_id": request_id_var.get("-"),  # ← 回顾 §1.1
        },
    )
```

响应：`HTTP 422 {"code": 42201, "message": "库存不足", "request_id": "abc12345"}`

注意 `request_id` 出现在异常响应中——它同样来自 `ContextVar`。前端或调用方可以凭这个 ID 在日志中定位完整链路。

**异常继承树**（在 `common/exceptions.py` 中定义，回顾 §1.2 异常码表）：

```
ShopException（基类，code=50000, http_status=500）
├── ValidationError     40001  400
├── AuthenticationError  40101  401
├── PermissionDeniedError 40301  403
├── NotFoundError        40401  404
├── BusinessError        42201  422
├── DatabaseError        50001  500
└── InternalError        50002  500
```

路由函数中只需要 `raise BusinessError("库存不足")`，其他全部自动。

> **`@app.exception_handler(ShopException)` 能捕获所有业务异常吗**：能。FastAPI 基于**继承链匹配**处理器（类似 `isinstance`）。`BusinessError` → `ShopException` → `Exception` 这个链上，第一个匹配的是 `ShopException` 的处理器——因此全部 7 个子类都被它捕获。项目中还有一个 `@app.exception_handler(Exception)` 作为兜底——未预期的非业务异常（如 `ZeroDivisionError`）由它处理，返回 50002。两个处理器注册顺序不影响匹配结果，FastAPI 总是优先匹配最具体的类型。

### 2.8 第七站：成功响应 —— 原路返回

下单成功，路由函数执行到最后一行：

- `RequestIDMiddleware` 在响应头中写入 `X-Request-ID: abc12345`

调用方收到完整的 HTTP 响应。

## 第3章 下一次请求：缓存加速

> 支线：追踪 `GET /c-endpoint/products/{id}` —— 一个典型的读请求，展示 Redis 如何让数据更快返回。

第 2 章的下单请求是"写操作"——涉及事务、锁、库存变更。但大多数用户请求是"读操作"——浏览商品列表、查看商品详情。本章以商品详情为例。

---

### 3.1 请求进入商品详情

```
GET /api/shop/c-endpoint/products/5
```

中间件链、认证（详情接口不强制登录）、Pydantic 校验等过程与第 2 章相同，此处略过。直接进入业务层：

```python
def get_product_detail(product_id: int) -> dict:
    cache_key = f"product:{product_id}"

    # 第一步：先查 Redis
    cached = get_cache(cache_key)
    if cached is not None:
        return cached                          # 命中 → 直接返回（1-3ms）

    # 第二步：Redis 未命中 → 查 PostgreSQL
    with get_cursor() as cur:
        cur.execute("SELECT ... WHERE p.id = %s AND p.status = 'on_sale'", (product_id,))
        row = cur.fetchone()
        if not row:
            raise NotFoundError("商品不存在")

    # 第三步：回写 Redis（10 分钟后自动过期）
    product = dict(row)
    set_redis(cache_key, product, ttl=600)
    return product
```

---

### 3.2 为什么要缓存

同一个商品详情可能在短时间内被几十个用户访问。每次查 PostgreSQL 需要 10-30ms，Redis 读取只要 1-3ms（数据全在内存中，单线程无锁竞争）。

| 场景 | 直接查 PostgreSQL | Redis 缓存 |
|------|-------------------|-----------|
| 商品详情 | ~10-30ms | ~1-3ms |
| 首页热门商品（5 条 JOIN 分类） | ~50-100ms | ~3-5ms |

---

### 3.3 Cache-Aside 策略

本项目采用最经典的缓存策略——Cache-Aside：

```
读路径：
  ① 先查 Redis → 命中 → 返回
  ② 未命中 → 查 PostgreSQL → 回写 Redis → 返回

写路径：
  ① 更新 PostgreSQL
  ② 删除受影响的 Redis key（不更新缓存，由下次读触发回写）
```

**为什么写操作是删除而非更新**：管理员编辑了商品价格——如果更新 Redis，你要在写操作中重新执行"查 DB + 序列化 JSON"。但这次写完之后可能根本没人访问——浪费。删掉 key，下次有人读时自然触发回写，用到了才重建。

项目中三个缓存 key：

| Key | 内容 | TTL | 为什么是这个 TTL |
|-----|------|-----|-----------------|
| `product:{id}` | 单商品详情 | 10 min | 详情变更不频繁 |
| `hot:products:list` | 最新 5 条 on_sale 商品 | 5 min | 首页高频，变化较快 |
| `categories:tree` | 二级分类树 | 30 min | 分类结构极少变动 |

---

### 3.4 缓存失效：B 端写操作时删什么

B 端管理员的操作会触发缓存删除（不是更新）：

| B 端操作 | 需删除的 Key | 原因 |
|----------|------------|------|
| 发布新商品 | `hot:products:list` | 新商品可能进入热门列表 |
| 编辑商品 | `product:{id}` + `hot:products:list` | 详情变了 + 列表中数据过时 |
| 上下架商品 | `product:{id}` + `hot:products:list` | 同上 |
| 分类增删改 | `categories:tree` | 分类结构变了 |

实现上，项目封装了 `delete_keys()` 函数使用 Redis Pipeline 批量发送删除指令——一次网络往返完成多个 DEL。

---

### 3.5 互斥锁：防止缓存击穿

**① 问题**：热门商品缓存刚好过期。在"过期时刻"到"新值写回"之间，来了 50 个请求——它们都发现缓存 miss，全都去查 DB。这意味着数据库瞬间承受 50 倍的查询压力。这叫做**缓存击穿**。

**② 概念**：用一把互斥锁保证同一时刻只有一个请求查 DB 并回写，其余请求等锁释放后直接读缓存。

```python
_cache_locks: dict[str, threading.Lock] = {}

def get_product_detail(product_id: int) -> dict:
    cache_key = f"product:{product_id}"

    # 第一次检查
    cached = get_cache(cache_key)
    if cached is not None:
        return cached

    # 获取该 key 的专属锁
    lock = _get_cache_lock(cache_key)
    with lock:
        # 双重检查：等锁期间可能已被其他线程写入
        cached = get_cache(cache_key)
        if cached is not None:
            return cached

        # 只有第一个拿到锁的线程查 DB
        product = _query_db(product_id)
        set_redis(cache_key, product, ttl=600)
        return product
```

> **为什么单线程 FastAPI 需要锁**：psycopg2 是同步驱动，FastAPI 的路由函数用 `def`（非 `async def`）时在线程池中执行——所以多请求确实并发在不同线程中。

---

### 3.6 TTL 自愈与降级

**TTL 自愈**：即使写操作删除缓存后，某个读请求在极短的时间窗口内回写了旧数据，TTL（5-10 分钟）到期后自动删除，下一次读拿到正确数据。更重要的是，下单时 `FOR UPDATE` 从 DB 直接读真实值——缓存不一致只会导致短暂页面展示偏差，不会产生错误订单。

**降级**：Redis 不可用时，跳过缓存直接查 DB。系统变慢但不崩溃：

```python
def get_cache(key: str):
    try:
        return json.loads(redis_client.get(key))
    except Exception:
        return None  # 降级：像缓存未命中一样，后续查 DB
```

---

### 3.7 本章小结

读请求和写请求的两条路径对比：

```
读请求（商品详情）：                  写请求（商品编辑）：
  Redis 命中 → 返回 (1-3ms)           UPDATE DB
     ↓ 未命中                           ↓
  获取互斥锁                           DEL product:{id}
     ↓                                 DEL hot:products:list
  查 DB → 回写 Redis → 返回              ↓
                                   下次读请求触发回写

降级：Redis 不可用 → 全部走 DB
自愈：脏数据在 TTL 到期后自动清理
```

## 第4章 请求之外

> 两个不经过 HTTP 请求的东西：应用启动时的资源初始化，和时钟驱动的定时任务。

### 4.1 Lifespan：启动时发生了什么

**`lifespan` 的本质**：FastAPI 提供的**应用生命周期钩子**——不处理请求，而是管理整个应用级全局资源的初始化与释放。启动时创建一次（数据库连接池、Redis 客户端、调度器），运行期间所有请求共享这些资源，关闭时统一释放。`yield` 之前是启动逻辑，`yield` 之后是关闭逻辑：

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    # ===== 启动时 =====
    setup_logger()              # §1.3 的 logging 配置生效
    init_pool()                 # 创建数据库连接池（§2.6）
    init_redis()                # 创建 Redis 客户端（§3.3）
    scheduler.add_job(          # 注册定时任务（§4.2）
        cancel_timeout_orders,
        trigger="interval",
        minutes=5,
    )
    start_scheduler()
    yield                       # ← 应用正常运行
    # ===== 关闭时 =====
    shutdown_scheduler()
    close_redis()
    close_pool()
```

`lifespan` 是项目所有基础设施资源的**统一挂载点**。你在前面章节中遇到的 `get_connection()`、`get_redis()`——它们能工作的前提，就是 `lifespan` 在这里调了 `init_pool()` 和 `init_redis()`。

---

### 4.2 定时任务：超时订单自动取消

**① 问题**：用户下单扣了库存，但 30 分钟不支付。库存被永久锁住。需要后台定期扫描并取消这类订单。

**② 概念**：APScheduler 是一个进程内定时任务库。`BackgroundScheduler` 在后台线程中运行，每隔固定时间触发指定函数。

```python
from apscheduler.schedulers.background import BackgroundScheduler

scheduler = BackgroundScheduler(
    timezone="Asia/Shanghai",
    job_defaults={
        "coalesce": True,          # 上次任务还在跑则跳过本次
        "max_instances": 1,        # 同一任务最多同时 1 个实例
        "misfire_grace_time": 60,  # 错过 ≤ 60s 补执行，超过跳过
    },
)
```

> **APScheduler 的四个组成部分**：
> - **触发器（Trigger）**：决定什么时候执行。本项目用 `interval`——每隔 5 分钟触发一次。APScheduler 还支持 `cron`（"每周三 9:00"）和 `date`（"2026-06-01 12:00 执行一次"）。
> - **任务存储（Job Store）**：存任务定义。本项目默认存内存——进程重启任务丢失，但 `lifespan` 在每次启动时重新注册，所以不丢。生产环境可改用数据库持久化。
> - **执行器（Executor）**：决定任务在哪个线程执行。`BackgroundScheduler` 默认用线程池——任务在线程池中执行，不阻塞主线程（FastAPI 的事件循环）。
> - **调度循环**：一个后台线程循环检查——"有没有任务的触发时间 ≤ 当前时间？有就扔给执行器执行。"

**③ 定时任务的核心逻辑**（与第 2 章手动取消订单共享同一个 `cancel_order_by_id` 函数）：

```
每 5 分钟触发
  ↓
SELECT id FROM orders WHERE status='pending' AND created_at < NOW() - 30 分钟
  ↓
无结果 → 等待下一轮
有结果 → 逐条处理（每条独立事务）：
    FOR UPDATE 锁定订单
    ↓
    二次校验 status 仍为 pending（防止与手动取消竞态）
    ↓
    UPDATE status = cancelled
    ↓
    回滚库存到 products（SQL 原子加法：SET stock = stock + quantity）
    ↓
    COMMIT
```

> **为什么逐条事务**：如果 10 条超时订单放在一个大事务中，其中一条失败导致整体 ROLLBACK，全部 10 条都没取消。逐条事务互不影响。

## 第5章 服务间调用

> AI 客服需要查询订单和物流数据，但它不应该直连 shop 数据库。它通过 HTTP 调用电商服务的内部接口。

### 5.1 为什么不让 AI 服务直连数据库

四个理由：
1. **安全**：AI 服务如果可以直接读 `shop` schema，任何配置错误都可能导致数据泄露
2. **耦合**：AI 服务需要知道表结构和字段名，任何变更都要同步改
3. **职责混乱**：AI 服务变成了"半个电商服务"
4. **业务边界**：`shop` schema 属于电商服务的业务域。AI 服务不应成为电商数据的"另一入口"。通过接口层隔离，电商服务可独立变更表结构、拆表、换数据库，只要接口不变，AI 服务不受影响——这是"低耦合"的体现

正确的做法：

```
用户问 → AI 服务 → HTTP GET /internal/orders?user_id=42 → 电商服务 → 查 DB → 返回数据
```

---

### 5.2 httpx：异步 HTTP 客户端

AI 服务是 FastAPI 应用（异步框架）。Python 最常用的 `requests` 库是同步的——调用 `requests.get()` 会阻塞事件循环。`httpx` 是支持 `async/await` 的替代：

```python
import httpx

async def get_user_orders(user_id: int) -> dict:
    async with httpx.AsyncClient(timeout=httpx.Timeout(10.0)) as client:
        try:
            resp = await client.get(
                f"{SHOP_INTERNAL_URL}/internal/orders",
                params={"user_id": user_id},
                headers={"X-Internal-Token": INTERNAL_API_TOKEN},
            )
            return resp.json()
        except Exception:
            return {"code": -1, "data": {}, "message": "电商服务不可用"}
```

关键点：
- `await` 不阻塞事件循环——在等网络 IO 期间其他请求继续处理
- `timeout=10.0` 设置 10 秒超时，电商服务无响应时不会永远等待
- 异常时返回降级响应，不抛出异常

---

### 5.3 内部接口认证

电商服务的 `/internal/*` 接口不对外开放。认证方式是固定 Token：

```python
# 电商侧校验
def verify_internal_token(x_internal_token: str = Header(..., alias="X-Internal-Token")):
    if x_internal_token != INTERNAL_API_TOKEN:
        raise AuthenticationError("无效的内部接口 Token")
    return True

# AI 服务侧携带
headers={"X-Internal-Token": INTERNAL_API_TOKEN}
```

Token 通过 `.env` 注入，两服务共享。

> **`X-Internal-Token` 是 JWT 吗**：不是。它只是一个固定字符串（如 `dev-internal-token`），从 `.env` 读取，校验就是一行字符串比对。没有 JWT 的 Header.Payload.Signature 结构、过期时间、签名验证——因为这些在服务间内部通信中不需要。两个服务在同一 Docker 网络中，信任关系是固定配置而非动态签发。
>
> **`Header(..., alias="X-Internal-Token")` 是 httpx 独有的吗**：不是，这是 **FastAPI 的参数声明机制**，与 httpx 完全无关。`...` 表示必填，`alias` 解决 HTTP 头名（含连字符）与 Python 变量名（不能含连字符）的映射——`x_internal_token` 对应 `X-Internal-Token`。AI 服务那边只是发了一个标准 HTTP 头 `headers={"X-Internal-Token": "..."}` ——curl、requests、httpx 任何 HTTP 客户端都能做到。

---

### 5.4 降级：电商不可用时

电商服务挂了，AI 客服不应该报技术错误给用户。每个查询函数在异常时返回 `{"code": -1, "message": "电商服务不可用"}`，Master Agent 检查 `code != 0` 时转为友好提示或纯 FAQ 回答——用户看到的是"抱歉，暂时无法查询订单信息"，不会看到技术细节。

## 第6章 容器化部署

> 所有东西打包进 5 个 Docker 容器，一个命令全部启动。

### 6.1 Docker Compose 解决什么问题

开发环境不一致的经典问题：你的电脑上 Python 3.10、PostgreSQL 16、Redis 7 都完美运行，换一台电脑全炸。Docker 把应用及其全部依赖打包成镜像，在任何安装了 Docker 的机器上运行结果一致。

Docker Compose 编排多个容器——本项目定义在 `docker-compose.yml` 中：

```
services:
  nginx          # 反向代理，统一入口
  shop-service   # 电商服务
  ai-service     # AI 客服服务
  postgres       # PostgreSQL 16 + pgvector
  redis          # Redis 7
```

`docker compose up -d` 一个命令，5 个容器全部启动。

---

### 6.2 容器间通信：服务名即主机名

Docker Compose 自动为每个 service 创建 DNS 记录。`shop-service` 容器内部可以直接用 `postgres:5432` 访问 PostgreSQL——Docker 内部的 DNS 把 service 名解析为容器的内网 IP。不需要知道 IP 地址（每次启动都可能变化）。

---

### 6.3 启动顺序：depends_on + healthcheck

`shop-service` 启动时 PostgreSQL 可能还没完成初始化（init.sql 建表），连接会失败。`depends_on` + `healthcheck` 解决这个问题：

```yaml
postgres:
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U user -d agent"]
    interval: 5s

shop-service:
  depends_on:
    postgres:
      condition: service_healthy   # ← 等待 PostgreSQL 健康检查通过
```

`pg_isready` 确认数据库进程在监听后才允许 shop-service 启动。

### 6.4 Nginx：反向代理

前端只需要知道一个入口 `http://localhost:83`，Nginx 根据 URL 路径转发：

```
浏览器 → Nginx (:83)
            ├── /api/shop/* → shop-service:8004
            └── /api/ai/*   → ai-service:8000
```

Nginx 负责：统一入口、路径路由、CORS 头设置。

**转发过程**（以 `POST /api/shop/c-endpoint/orders` 为例）：

1. 浏览器访问 `http://localhost:83/api/shop/c-endpoint/orders`——用户只看到 `localhost:83` 这一个地址
2. Nginx 检查 URL 前缀——`/api/shop` 开头的请求，剥离 `/api/shop` 前缀后转发到 `shop-service:8004`
3. shop-service 收到 `POST /c-endpoint/orders`——路由匹配到订单 Router 的 `@router.post("")`
4. 同样，`/api/ai/*` 开头的请求转发到 `ai-service:8000`

| 用户访问 | Nginx 转发到 | 后端路由 |
|----------|------------|---------|
| `GET /api/shop/c-endpoint/products` | `shop-service:8004/c-endpoint/products` | `c_endpoint product router` |
| `POST /api/shop/c-endpoint/orders` | `shop-service:8004/c-endpoint/orders` | `c_endpoint order router` |
| `GET /api/shop/internal/orders?user_id=42` | `shop-service:8004/internal/orders` | `internal router` |
| `POST /api/ai/chat` | `ai-service:8000/ai/chat` | `chat router` |

### 6.5 数据持久化：volumes

容器删除后，数据库文件默认会丢失。Volume 把宿主机目录挂载到容器内，容器删除数据保留：

```yaml
volumes:
  pgdata:      # 声明 volume

postgres:
  volumes:
    - ./init.sql:/docker-entrypoint-initdb.d/init.sql  # 文件：首次启动自动建表
    - pgdata:/var/lib/postgresql/data                  # volume：数据持久化
```

`init.sql` 在 PostgreSQL 容器首次启动时自动执行——完成建表（`CREATE TABLE shop.users ...`）和种子数据插入（预置管理员账号）。
