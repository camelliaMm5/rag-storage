## 三、数据库与连接池

### 3.1 连接池参数

| 参数 | 值 | 理由 |
|------|-----|------|
| 连接池类型 | ThreadedConnectionPool | psycopg2 内置，零额外依赖 |
| minconn | 2 | 保证至少 2 个空闲连接应对突发请求 |
| maxconn | 10 | 单实例部署，覆盖 C 端 + B 端 + Scheduler 并发 |

### 3.2 查询规范

- 所有操作均在 `shop` schema 内，无需跨 schema 查询
- 连接池所有连接统一设置 search_path 为 `shop`，避免连接间行为不一致

### 3.3 事务管理约定

| 场景 | 策略 | 说明 |
|------|------|------|
| 普通查询 | 自动提交 | psycopg2 默认行为 |
| 下单 | 手动事务 | 锁库存→扣减→创订单→清购物车，任一失败 ROLLBACK |
| 支付 | 手动事务 | 锁订单→校验→更新→写支付记录 |
| 取消订单 | 手动事务 | 锁订单→校验→更新→回滚库存 |
| 定时任务取消 | 每单独立手动事务 | 互不影响 |

## 四、中间件设计

### 4.1 RequestIDMiddleware

**解决的问题**：每个 HTTP 请求需要一个唯一标识贯穿所有日志，便于在 `docker logs` 中追踪单次请求全链路。

**机制**：从请求头 `X-Request-ID` 读取（缺失则自动生成 8 位 UUID），存入 Python `ContextVar`（协程安全），响应头回写。

### 4.2 RequestLogMiddleware

**解决的问题**：记录每个请求的方法、路径、状态码、耗时，统一日志格式，取代在各路由函数中手写 log 语句。

**机制**：拦截请求 → 记录 `time.time()` → 执行后续中间件/路由 → 计算耗时 → 通过 Python Logger 输出到 stdout → Docker 捕获。

### 4.3 全局异常处理器

**解决的问题**：业务层抛出的 `ShopException` 子类自动映射为统一 JSON 响应 `{code, message, request_id}`，避免在路由中手写 try/except。

**异常码映射**（继承自需求文档 14.4 节）：

| 异常类 | code | HTTP 状态码 |
|--------|------|------------|
| ValidationError | 40001 | 400 |
| AuthenticationError | 40101 | 401 |
| PermissionDeniedError | 40301 | 403 |
| NotFoundError | 40401 | 404 |
| BusinessError | 42201 | 422 |
| DatabaseError | 50001 | 500 |
| InternalError | 50002 | 500 |

### 4.4 CORS

开发阶段允许所有来源。FastAPI 通过 `CORSMiddleware` 配置。

## 五、认证与鉴权

### 5.1 JWT 设计

| 字段 | 来源 | 说明 |
|------|------|------|
| `user_id` | shop.users.id | 用户唯一标识 |
| `email` | shop.users.email | 登录凭证 |
| `role` | shop.users.role | `user` / `admin` |
| `exp` | 签发时间 + 24h | 过期时间 |

- 签发：HS256 算法，Secret 来自环境变量 `JWT_SECRET`
- 验证：FastAPI 依赖注入 `Depends(get_current_user)` 从 `Authorization: Bearer <token>` 解析

### 5.2 角色权限矩阵

| 角色 | C 端接口 | B 端接口 | `/internal/*` |
|------|---------|---------|---------------|
| user | 仅操作自身数据 | 拒绝 | 拒绝 |
| admin | 可访问 | 全部权限 | 拒绝 |
| AI 服务 | 拒绝 | 拒绝 | `X-Internal-Token` 认证 |

### 5.3 内部接口认证

外部 AI 客服服务调用电商 `/internal/*` 接口时，携带 HTTP Header：

```
X-Internal-Token: ${INTERNAL_API_TOKEN}
```

Token 值通过 `.env` 注入，电商服务与 AI 服务共享。接口侧通过依赖注入校验，不通过返回 401。

## 六、定时任务

### 6.1 方案选型

| 方案 | 评价 | 选择 |
|------|------|------|
| APScheduler | 零外部依赖、Python 原生、轻量 | **选用** |
| Redis 过期回调 | 依赖 Redis keyspace notification，可靠性一般 | 淘汰 |
| 数据库 cron 脚本 | 与代码分离，不易维护 | 淘汰 |

### 6.2 配置参数

| 参数 | 值 | 理由 |
|------|-----|------|
| 扫描间隔 | 5 分钟 | Demo 场景避免频繁 DB 查询 |
| 超时阈值 | 30 分钟 | 足够用户完成支付操作 |
| misfire_grace_time | 60 秒 | 调度器短暂故障恢复后，错过时间 ≤ 60 秒则补执行，超过则跳过 |

> **misfire_grace_time 说明**：与订单超时无关，处理的是 APScheduler 自身故障恢复场景。例如 10:00 应触发扫描但恰好服务重启，10:02 恢复时差了 2 分钟 > 60 秒，则跳过本次，等下一个周期 10:05。避免服务重启后堆积大量补偿执行。

### 6.3 设计要点

- **逐条事务**：不把全部超时订单放一个大事务，一单失败不影响其他
- **二次校验**：`FOR UPDATE` 锁行后仍校验 `status='pending'`，防止与手动取消竞态
- **库存回滚用 SQL 原子加法**：`SET stock = stock + quantity`，不在应用层计算
- **部分索引**：`idx_orders_pending_time WHERE status='pending'` 仅索引 pending 行，扫描效率高
- **生命周期**：Scheduler 在 FastAPI `lifespan` 中启动，应用关闭时 `shutdown(wait=False)`

## 七、日志方案

### 7.1 整体架构

```
Python 应用
    │
    │ logging.StreamHandler(sys.stdout)
    ▼
容器 stdout ──→ Docker Log Driver (json-file) ──→ /var/lib/docker/containers/*/*.log
```

**核心原则**：应用只往 stdout 打日志，存储和轮转让 Docker 处理。切换日志后端（json-file → Loki）时应用代码零改动。

### 7.2 Logger 与 Docker 分工

| 职责 | Python Logger | Docker |
|------|-------------|--------|
| 日志生成（格式化、级别过滤） | 负责 | — |
| 日志输出（写到哪） | stdout | 捕获 stdout |
| 日志存储（落盘） | 不做 | json-file driver |
| 日志轮转 | 不做 | max-size=10m, max-file=3 |
| 日志查看 | — | `docker logs` 命令 |
| 日志聚合 | — | Loki/Fluentd/ELK（后续扩展） |

### 7.3 日志格式

```
时间 | 级别 | request_id | 模块:行号 | 消息正文
```

- `request_id` 由 `RequestIDFilter` 从 `ContextVar` 注入每条日志
- 第三方库日志抑制：APScheduler、redis-py 的 DEBUG 日志提升为 WARNING
- 定时任务日志使用 `[Scheduler]` 前缀

### 7.4 Docker 日志轮转

```yaml
logging:
  driver: json-file
  options:
    max-size: "10m"
    max-file: "3"
```

## 八、内部接口

> 本章定义的接口是电商服务对外部 AI 客服服务暴露的查询契约。AI 服务为外部独立项目，通过以下接口获取实时业务数据，不直连数据库。

### 8.1 接口清单

| 端点 | 方法 | 描述 | 调用方 |
| --- | --- | --- | --- |
| `/internal/orders` | GET | 查询用户订单列表 | AI 客服 |
| `/internal/orders/{order_id}` | GET | 查询订单详情 | AI 客服 |
| `/internal/logistics` | GET | 查询用户物流信息 | AI 客服 |
| `/internal/after-sales` | GET | 查询用户售后申请 | AI 客服 |
| `/internal/products/search` | GET | 关键词搜索商品 | AI 客服 |
| `/internal/products/{product_id}` | GET | 查询商品详情 | AI 客服 |
| `/internal/users/{user_id}` | GET | 查询用户基本信息 | AI 客服 |

### 8.2 认证方式

```
Header: X-Internal-Token: ${INTERNAL_API_TOKEN}
```

路由通过 FastAPI 依赖注入 `Depends(verify_internal_token)` 全局校验，不通过返回 401。

### 8.3 返回格式约定

```json
{"code": 0, "data": {...}, "message": "success"}
```

- `code=0` 成功，非 0 异常
- 列表接口额外返回 `page` / `size` 分页信息
- 默认每页 20 条

### 8.4 设计原则

- **只读**：所有内部接口仅提供查询
- **过滤**：订单/物流/售后按 `user_id` 过滤，确保数据隔离
- **容错**：电商不可用时 AI 服务应降级为纯 FAQ 回答
- **分页**：列表接口支持 `page` / `size`

## 九、缓存方案

### 9.1 为什么用 Redis

| 对比维度 | 直接查 PostgreSQL | Redis 缓存 |
| --- | --- | --- |
| 商品详情响应时间 | ~10-30ms | ~1-3ms |
| 首页热门商品（5 条 JOIN 分类） | ~50-100ms | ~3-5ms |
| 数据库并发压力 | 每次请求打到 DB | 命中缓存时跳过 DB 查询，降低数据库负载 |

### 9.2 缓存架构（Cache-Aside）

```
读请求 → Redis 命中? → 是 → 返回
              ↓否
          查 PostgreSQL → 回写 Redis → 返回

写请求 → 更新 PostgreSQL → 删除 Redis key（不更新缓存）
```

**读路径**：优先查 Redis → 命中返回 → 未命中查 DB → 回写 Redis → 返回。

**写路径**：先更新 DB → 再删除受影响的 Redis key。不直接更新缓存，由下次读请求触发回写。

### 9.3 缓存 Key 设计

| Key | 数据类型 | 内容 | TTL | 失效触发 |
| --- | --- | --- | --- | --- |
| `hot:products:list` | String (JSON) | 最新 5 条 on_sale 商品（含分类名），按创建时间倒序 | 5 min | 商品发布/编辑/上下架 |
| `product:{id}` | String (JSON) | 单商品完整信息 | 10 min | 该商品编辑/上下架 |
| `categories:tree` | String (JSON) | 二级分类树 | 30 min | 分类增删改 |

### 9.4 缓存失效矩阵

| B 端操作 | 需删除的 Redis Key | 原因 |
| --- | --- | --- |
| 发布新商品 | `hot:products:list` | 新商品可能进入热门列表 |
| 编辑商品 | `product:{id}` + `hot:products:list` | 详情变更 + 列表中数据过时 |
| 上架商品 | `product:{id}` + `hot:products:list` | 状态变更 + 应出现在列表 |
| 下架商品 | `product:{id}` + `hot:products:list` | 不可见 + 应从列表移除 |
| 分类增删改 | `categories:tree` | 结构变更 |

### 9.5 删除缓存 vs 更新缓存

| 策略 | 做法 | 评价 |
| --- | --- | --- |
| 更新缓存 | DB 写完后 `SET` 新值 | 写后若无人读，浪费内存；热门列表涉及 JOIN+排序，写操作中重算代价高 |
| **删除缓存（选用）** | DB 写完后 `DEL` 对应 key | 仅被读取时才回写，不浪费；写操作轻量 |

### 9.6 并发竞态窗口

```
时间线：  ──[DB UPDATE]──[Redis DEL]──→
                            ↑
                    此窗口内有读请求→读到旧缓存→回写→覆盖 DEL
```

**决策**：单次删除，不引入延迟双删。理由：
- 单机部署下窗口为毫秒级，命中概率极低
- 缓存不一致仅导致短暂展示偏差（价格/库存显示滞后），不会产生错误订单——下单时 `FOR UPDATE` 重新从 DB 读取真实值并校验
- 即使脏数据残留，TTL（5-10 分钟）自动过期清理

### 9.7 降级策略

| 场景 | 处理 |
|------|------|
| Redis 连接失败（读） | 跳过缓存，直接查 DB，记录 WARNING |
| Redis 连接失败（回写） | 忽略，不影响业务响应；下次请求仍 miss，多查一次 DB |
| Redis 连接失败（删除） | 忽略；缓存依赖 TTL 自动过期清理 |
| Redis 持续不可用 | 所有请求走 DB，体验下降但系统可用 |

### 9.8 设计决策记录（已确认）

| 编号 | 决策 | 结论 |
|------|------|------|
| D1 | 更新粒度 | 整体 key 删除，不做部分更新。热门列表仅 5 条，整体重建代价可忽略 |
| D2 | 并发竞态 | 单次删除，不引入延迟双删。DB 事务兜底 + TTL 自愈（详见 6.6 节） |
| D3 | 批量失效 | 接口接受 id 列表，Redis Pipeline 合并发送，单个是批量的特例 |
| D4 | 回写失败 | 仅记日志、不重试。缓存依赖 TTL 自动过期清理，降级查 DB（详见 6.7 节） |
| D5 | 缓存开关 | 不需要。缓存作为基础设施直接集成，不预留开关 |

## 十、架构决策记录 (ADR)

> 记录项目关键架构决策及理由，供后续开发与评审参考。

| 编号 | 决策 | 理由 |
|------|------|------|
| ADR-1 | 使用单一 `shop` schema | 当前表量少（8 张），双 schema 增加开发摩擦；B/C 权限隔离在应用层通过 JWT role 实现，无需数据库层面隔离 |
| ADR-2 | 管理员账号通过 init.sql 预置种子数据 | 一键部署即用，预置 bcrypt 哈希 |
| ADR-3 | 超时阈值 30min，扫描间隔 5min | Demo 可快速验证 |
| ADR-4 | 热门商品按创建时间倒序 | 实现简单，预留销量排序扩展点 |
| ADR-5 | Redis 不做持久化 | 数据可从 DB 完全重建，节省资源 |
| ADR-6 | 内部 API 固定 Token + .env 注入 | Demo 足够，生产建议 Vault 轮换 |
| ADR-7 | 日志 json-file + 轮转，暂不对接 Loki | 满足调试需求，集中日志后续扩展 |
| ADR-8 | 暂不设置 API 速率限制 | Demo 无恶意流量风险 |
| ADR-9 | 单机部署，不考虑分布式 | 教学 Demo 定位 |

### 12.1 环境变量完整清单

```bash
# 数据库
DATABASE_URL=postgresql://user:1234@postgres:5432/agent

# JWT
```
