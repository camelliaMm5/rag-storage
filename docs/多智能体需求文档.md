# 多智能体客服系统扩展 —— 需求讨论纪要

## 项目背景

基于已完成的 ReAct Agent 智能客服（见 [guide_agent_extension_clean.md](guide_agent_extension_clean.md)），将其扩展为多智能体协同系统。

**核心约束**：学生可复现的 demo，感受企业级 Agent 开发的同时保持难度适中。细节处理和复杂情况可提示但不必须完成。

---

## 1. 总体决策概览

| 维度 | 决定 |
|------|------|
| 多智能体架构 | 层级式：1 Master Agent + 3 Sub-Agent + FAQ 兜底 |
| 业务场景 | 电商（订单查询 / 物流跟踪 / 售后处理） |
| 多智能体框架 | LangGraph |
| 模拟数据 | 通过统一数据访问层获取，具体实现由开发者决定 |
| 项目结构 | 纯后端 Python 项目，通过 HTTP API 对外服务 |
| 后端通信 | 流式 SSE，打字机效果 |
| Agent 推理可见性 | 通过 SSE 分阶段推送中间推理过程（路由决策 → 执行中 → 汇总） |
| 用户数据隔离 | 会话绑定 `user_id`，仅允许查询自身数据 |
| 身份认证 | JWT Bearer Token 认证，`user_id` 从 Token 中提取 |
| 演示方式 | 4 个预设场景 + 边界条件验证（用户数据隔离），通过 API 调用验证 |
| 鲁棒性 | 基础兜底策略，维持系统基本稳定 |

---

## 2. 多智能体架构

### 2.1 层级式架构

```
用户 → Master Agent（意图路由 + 结果汇总）
           ├─→ OrderAgent（订单查询）     → 模拟订单数据
           ├─→ LogisticsAgent（物流跟踪） → 模拟物流数据
           └─→ AfterSaleAgent（售后/退款） → 模拟售后数据 + FAQ 知识库
                    ↓
           Master Agent（汇总）→ 用户
```

- **Master Agent**：意图识别 → 路由分发 → 收集 Sub-Agent 结果 → 汇总为自然语言回复
- **Sub-Agent**：各自负责一个业务领域，拥有独立的工具和数据源，内聚可独立测试
- **FAQ 检索**（已有）作为 Master 的通用兜底工具，处理三个 Sub-Agent 覆盖范围外的问题
- **Agent 执行韧性**：Agent 执行过程中不应因异常或错误而中断，需具备自我恢复能力

### 2.2 Sub-Agent 职责与数据

| Sub-Agent | 职责 | 数据来源 |
|-----------|------|----------|
| OrderAgent | 查询订单状态、订单详情 | 模拟订单数据 |
| LogisticsAgent | 查询快递进度、预计送达 | 模拟物流数据 |
| AfterSaleAgent | 退款/退货/换货流程与进度 | 模拟售后数据 + FAQ 知识库 |

### 2.3 模拟数据

各 Sub-Agent 通过统一的数据访问层获取模拟业务数据：
- 能展示"查到"和"查不到"两种结果
- 边界条件：用户只能查询自身数据（通过 `user_id` 过滤），查不到他人数据
- 具体实现方式（本地 JSON / 内存字典 / 外部 Mock 服务）由开发者自行决定，后续可切换为真实 API 调用

## 3. 多智能体框架

本系统使用 **LangGraph** 作为多智能体编排框架。

### 3.1 框架对比

| 维度 | LangGraph | 自研扩展 |
|------|-----------|----------|
| 路由控制 | 精确到每条边，手动建图 | 完全可控 |
| 上手难度 | 中高，需理解状态图 | 最低 |
| 灵活性 | 极高，图结构任意定制 | 最高 |
| 代码量（估） | ~80 行核心逻辑 | ~150 行 |
| 调试体验 | LangSmith 可视化追踪 | 靠 print |
| 社区认可度 | LangChain 主推，高 | — |
| 对现有项目适配 | 图结构迁移 ReAct 循环自然 | 在现有基础上扩展 |

## 4. 项目结构

```
project/
├── main.py                     ← FastAPI 入口 + 依赖装配
├── apps/                       ← API 路由
│   └── customer_service/
├── domain/                     ← Agent 业务逻辑
│   └── customer_service/
├── tools/                      ← 工具层
├── llm/                        ← LLM 调用封装
├── utils/                      ← 通用层（会话管理等）
├── infrastructure/             ← 数据持久化
├── data/                       ← 向量数据库持久化
├── docker-compose.yml
└── requirements.txt
```

本项目为纯后端 Python 服务，通过 HTTP API 对外提供接口，可独立部署。

> **外部依赖**：订单/物流/售后查询依赖外部 shop-service（地址通过 `SHOP_INTERNAL_URL` 环境变量配置），部署时需确保该服务可用或替换为本地 Mock 实现。

---

## 5. 流式通信

- 后端通过 SSE（Server-Sent Events）分阶段推送状态
- LLM 边生成边推送，实现流式输出
- 中间推理过程（Master 路由决策 → Sub-Agent 执行中 → 结果汇总）通过 SSE 事件暴露给调用方

---

## 6. 用户数据隔离

- 会话创建时绑定 `user_id`，后续请求通过 `conversation_id` 关联
- 后端根据会话关联的 `user_id` 过滤数据
- Sub-Agent 查询模拟数据时，仅返回当前用户的数据

---

## 7. 鲁棒性

基础兜底策略：
- 意图识别失败 → 走 FAQ 兜底
- Sub-Agent 执行异常 → 返回友好错误提示，不崩溃
- 超出知识/数据范围 → 明确告知用户，不做幻觉回答

---

## 8. 演示场景

### 8.1 预设场景（验收标准）

| # | 场景 | 用户输入 | 预期路由 |
|---|------|----------|----------|
| 1 | 订单查询 | "我的订单 #12345 发货了吗？" | Master → OrderAgent → 汇总 |
| 2 | 物流跟踪 | "快递 SF1234 到哪了？" | Master → LogisticsAgent → 汇总 |
| 3 | 售后咨询 | "我买的衣服不合身，怎么退货？" | Master → AfterSaleAgent + FAQ → 汇总 |
| 4 | 兜底测试 | "今天天气怎么样？" | Master → 意图识别失败 → FAQ 兜底 |

### 8.2 边界条件

- 用户 A 无法查询到用户 B 的订单数据
- 用不同用户名分别验证数据隔离

### 8.3 验收标准

1. 一键启动（`docker-compose up`）
2. 连续走完 4 个预设场景，每个场景路由到正确的 Sub-Agent
3. 流式输出，对话过程中可见 Agent 中间推理状态（通过 SSE 事件）
4. 用户数据隔离生效
